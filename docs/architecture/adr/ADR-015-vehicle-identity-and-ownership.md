# ADR-015: Vehicle identity and ownership

- ID: ADR-015
- Date: 2026-07-20
- Status: Accepted (design), Not yet implemented
- Context: The Laravel PoC's first architecture bug conflated vehicle identity with current ownership in one mutable `customer_id` column, causing a real constraint violation when a VIN needed to move to a new customer. Checked against optimus-server: `_ensure_unique_active_vin()` in `app/vehicle_store.py` already scopes VIN uniqueness correctly per-tenant (`uq_vehicles_owner_active_vin`, a partial index excluding archived vehicles) — better than the PoC's first draft — but `VehicleUpdate` (`app/models.py:785`) does not expose `customer_id` as updatable at all, meaning optimus-server has no ownership-reassignment capability whatsoever, not a live bug. Full detail in `docs/architecture/STACK-DECISION.md` §4.
- Decision: When this capability is needed, model ownership as a `vehicle_customers`-style history table (vehicle_id, customer_id, effective_from, effective_to nullable) with exactly one open row per vehicle enforced by a partial unique index at the database level, not a mutable `customer_id` column. A `reassign_vehicle_owner()` store function closes the current row and opens a new one, emitting a `VehicleOwnershipEvent` mirroring the existing `EstimateApprovalEvent`/`WorkOrderStatusEvent` audit pattern. Estimates and work orders keep the `customer_id` they were created under permanently, independent of later reassignment.
- Alternatives considered: keep `customer_id` as a single replaceable column and simply allow updating it (rejected — loses ownership history, and any past estimate's "who authorized this" attribution would need to be captured redundantly elsewhere instead of once, correctly, at creation time); make ownership a first-class field on every business record instead of on the vehicle (rejected — duplicates the same fact everywhere instead of deriving it from one history table).
- Consequences: additive only — a new table and a new function, no changes to any existing `Vehicle.customer_id` read path. No current data-loss or corruption risk is being fixed; this closes an absent capability, not an active defect, so it carries no urgency.
- Files affected (when implemented): a new Alembic migration for `vehicle_customers`, `app/vehicle_store.py` (new `reassign_vehicle_owner()` function), `app/db_models.py`, a new `VehicleOwnershipEvent` model, and tests mirroring the PoC's `tests/Feature/VehicleOwnershipTest.php` cases (same-VIN-different-shop allowed, same-VIN-same-shop dedupes, reassignment preserves history and estimate attribution, reassignment is audited, database-level constraints enforced).
- Revisit if: a real shop workflow needs vehicle-reassignment before this reaches the top of the backlog, in which case it should be pulled forward rather than left until "steps 2-3 are stable" per the roadmap in `docs/architecture/STACK-DECISION.md` §7.
- Scope note: explicitly **not** part of Phase 1 (the security kernel). This ADR exists so the design isn't lost, not to schedule the work now.
