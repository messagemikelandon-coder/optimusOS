# ADR-014: Final stack and deployment

- ID: ADR-014
- Date: 2026-07-20
- Status: Accepted
- Context: A Laravel 12 + Livewire 4 proof-of-concept (`/home/dejake/optimus-laravel-poc/`) was built and verified to evaluate whether OptimusOS should migrate off FastAPI. Full analysis in `docs/architecture/STACK-DECISION.md`. This reaffirms and extends the earlier ADR-001 in `docs/context/DECISIONS.md` (2026-07-01, "Retain FastAPI as the backend until an approved architecture decision changes it") now that the alternative has actually been evaluated, not just deferred.
- Decision: Retain FastAPI + SQLAlchemy + PostgreSQL + Alembic as the canonical backend and the existing static-JS frontend, incrementally improved. Do not introduce Laravel/Livewire/PHP into the production deployment. Simplify the deployment from 5 containers to 3 in the same motion: FastAPI serves `app/static/` directly (it already mounts `/static`) instead of a separate nginx container, and the `worker` service is removed — verified to currently do nothing but a 60-second TCP heartbeat loop, with nothing anywhere in the codebase enqueuing to it.
- Alternatives considered: a phased Laravel migration (rejected — normalized for feature coverage, the PoC covers ~17% of domain breadth at shallower depth than production in nearly every domain it touches; true parity is estimated at 20-30x the PoC's current volume, i.e., a from-scratch rebuild of a comparably-sized system); a full Laravel rebuild (rejected — dominated by both other options on every row of the weighted matrix in `docs/architecture/STACK-DECISION.md` §2).
- Consequences: `app/main.py`'s 178 endpoints get split into `APIRouter` modules matching the existing 28-store grouping (mechanical, zero-behavior-change) to give new Command Center/plan-executor endpoints (ADR-017) a home other than a 5,105-line file. The `worker` and `frontend` (nginx) services are removed from `docker-compose.yml`; the DB-backed-queue re-introduction trigger (an async plan-execution step >2-3s, or scheduled maintenance jobs) is recorded here so removal isn't mistaken for a permanent architectural stance against background jobs.
- Files affected: `app/main.py`, `docker-compose.yml`, `scripts/optimus_worker.py`, `ops/nginx/default.conf`, `docs/context/DECISIONS.md`, `docs/context/CURRENT_STATE.md`.
- Revisit if: a future, separately-approved architecture review finds the application has become structurally unstable, or a real background-job need reopens the worker-service question.
