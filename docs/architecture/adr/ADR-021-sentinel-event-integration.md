# ADR-021: Sentinel event integration

- ID: ADR-021
- Date: 2026-07-20
- Status: Accepted (design), Not yet implemented
- Context: "Optimus Sentinel" is referenced as a target observability/coordination capability but does not exist anywhere in the current codebase (confirmed by a full-repo search — no references found in application code, docs, or `AGENTS.md`/`CLAUDE.md`). `app/observability.py` provides per-request JSON logging with a request-id `ContextVar`; `app/security_events.py` provides a narrow auth/billing security-event enum. Neither currently emits the tenant/user/request/workflow/agent-run/service/action/result/error/deployment-version/timestamp shape a real Sentinel integration needs.
- Decision: Sentinel integrates through structured events, middleware, and registered playbooks — never through premature microservices, and never through autonomous action. Initial scope is strictly observe, organize, correlate, alert, and recommend. Sentinel must never execute arbitrary AI-generated shell commands or modify production state outside a registered, human-approved playbook. This ordering is a hard sequencing constraint: the security kernel (ADR-020/Phase 1) must ship and pass its acceptance tests first; structured event emission (observation only) comes second; any action capability, gated by registered playbooks and explicit approval per the same one-approval-per-plan model as ADR-017, comes only after step two has run in production long enough to validate its event data is trustworthy.
- Alternatives considered: building Sentinel as a separate microservice from the start (rejected per the original product direction — "do not let Sentinel requirements turn the application into premature microservices"); giving Sentinel action capability from day one to accelerate value (rejected — explicitly the sequencing this ADR exists to prevent; security and trustworthy observation must come first).
- Consequences: Phase 1 (ADR-020) should add event *hooks* (extension points structured event emission can later attach to) without adding autonomy — e.g., every store function and the future plan-executor should have one clear place a structured event could be emitted from, even before Sentinel itself consumes it.
- Files affected (when implemented): `app/observability.py` (extended field shape), `app/security_events.py` (extended taxonomy), a new Sentinel event-emission call in each store function and the plan-executor (ADR-017), and eventually a registered-playbook definition mechanism for any future action capability.
- Revisit if: production experience after the observation-only phase reveals the event shape is insufficient for a specific alerting/correlation need — that should drive a schema extension, not a reason to skip ahead to action capability.
