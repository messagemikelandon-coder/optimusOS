# ADR-020: Security-kernel integration

- ID: ADR-020
- Date: 2026-07-20
- Status: Accepted — this is Phase 1 of the approved roadmap, in progress.
- Context: The approved roadmap (`docs/architecture/STACK-DECISION.md` §7) requires a reusable security kernel to exist and pass its own acceptance tests before any prompt/manual shared-execution work (ADR-017) or Sentinel integration (ADR-021) begins. optimus-server already has working authentication (`app/auth.py`, `app/account_security_store.py`), authorization/role dependencies (`OwnerAuthContextDep`/`SupportAuthContextDep`/etc.), tenant scoping (ADR-019), audit-adjacent event types (`EstimateApprovalEvent`, `WorkOrderStatusEvent`, `security_events.py`), rate limiting (`app/rate_limit.py`, 6 Redis-backed sliding-window instances), API-key handling (`app/openai_key_info.py`), and structured JSON logging (`app/observability.py`) — all of it built separately, over time, by different phases of work.
- Decision: Extract, consolidate, and harden the existing pieces into one reviewable, testable module. Do not redesign any mechanism (no new cryptography, password hashing, token format, or auth model) and do not remove any existing capability. Full inventory, gap ranking, and commit sequence are in `docs/architecture/PHASE1-SECURITY-KERNEL-PLAN.md`.
- Alternatives considered: building Sentinel observation first and deferring security consolidation (rejected — explicitly forbidden by the approved roadmap ordering: security must precede broad Sentinel autonomy); rewriting auth/authz from scratch instead of consolidating (rejected — no defect has been found that requires a redesign; the existing mechanisms work and are tested, they are simply spread across modules without one canonical entry point).
- Consequences: every capability in scope must end this phase with (a) exactly one implementation location, (b) a test proving other modules call it rather than re-implementing it, and (c) no behavior change to any existing endpoint's actual authorization/rate-limit/validation outcome. This is a consolidation phase, not a policy change.
- Files affected: see `docs/architecture/PHASE1-SECURITY-KERNEL-PLAN.md` for the full file list, database/API impact, and commit sequence.
- Revisit if: the inventory phase finds an actual security defect (not just duplication) requiring a behavior change — that must be called out explicitly and separately from the consolidation work, with its own test proving the fix and without being used to justify a broader redesign.
