# ADR-022: Operating mode and subscription tier as separate, service-resolved axes

- ID: ADR-022
- Date: 2026-07-20
- Status: Accepted (design), not yet implemented
- Context: OptimusOS must support three operating modes — Solo (owner-operator, no bays/roster/scheduling-complexity), Mobile Field (on-site service, travel/drive-out context, field-first workflows), and Shop (service desk, bays, technicians, scheduling, inventory, approvals) — plus a Technician role workspace, without forking the codebase or duplicating business logic per mode. The groundwork already exists but is informal: four router docstrings (`app/api/routers/bays.py:13-15`, `working_hours.py:17-22`, `schedule_blocks.py:17-25`, `appointments.py:27-33`, all Phase 2C, 2026-07-20) already classify routes against "Solo Mode"/"Shop Mode"/"Mobile Field Mode"/"Technician Mode" in comments only — "routing only, no gating added here." No `Shop.mode` column, enum, or dependency exists. Separately, `ShopSubscription.tier` (`app/db_models.py:2363-2365`) already implements three real, priced, seat-based tiers — `solo`, `team`, `shop` (`app/subscription_store.py:46-50`) — gating only technician-seat count today. The word "Solo" and the word "Shop" each name both a proposed operating mode and an existing subscription tier; these are unrelated axes that happen to share English words, and this ADR treats that collision as a naming risk to manage, not a reason to invent different mode names (the task instruction is explicit: do not invent tier names, and the required mode names are Solo/Mobile Field/Shop/Technician).
- Decision:
  1. **Mode and tier are separate columns on separate concerns.** Operating mode (`Shop.operating_mode`, proposed values `solo` / `mobile_field` / `shop`) shapes which workflows and navigation are relevant to how a shop operates. Subscription tier (`ShopSubscription.tier`, existing values `solo` / `team` / `shop`, unchanged) grants capabilities (seat count, feature depth) in exchange for payment. A shop's mode does not imply its tier, or vice versa: a one-person mobile detailer can be on the `team` tier (extra seats for a part-time helper), and a five-bay shop can run the `solo` tier if it under-hires seats. No code may infer one axis from the other.
  2. **One backend capability-resolution service is the only place either axis is evaluated.** Modeled directly on the existing tenant-boundary precedent (`effective_shop_id()`, ADR-019) and the existing shared-store-layer discipline (ADR-016): a single function/dependency resolves `(operating_mode, tier, role, per-shop overrides) -> ShopCapabilities`, called by routes, store functions, the manual UI (via a read endpoint, not hardcoded per-view assumptions), and — when ADR-017's plan-executor is built — the AI action path, using the exact mechanism ADR-016 already mandates for that future work ("must call existing store functions... must never reimplement tenant scoping itself"; the same sentence now applies to capability scoping).
  3. **UI hiding is never the enforcement mechanism.** The existing `data-owner-only`/`data-technician-only`/`data-support-only` nav pattern (`app/static/app.js:316-337`) is a real, working precedent for *conditional navigation*, but it is driven by role only and has no backend-enforced counterpart requirement today because every gated nav item already maps to a route already gated by an owner/technician auth dependency. Mode- and tier-driven UI must follow the same rule going forward: a nav item or form field hidden for Solo mode must correspond to a route/store call that independently rejects the same action if attempted directly, never client-side-only concealment.
  4. **Hidden data is never deleted.** Switching a shop's mode changes what is *shown as relevant*, never what is stored. A Shop-mode shop that switches to Solo keeps its bays and technician records; they become inactive-by-mode, not deleted, exactly like the existing `is_archived`/soft-delete pattern already used throughout the schema (`Bay.is_archived`, `Technician` archive routes, etc.).
  5. **This ADR authorizes design only.** No migration, route, dependency, or UI change ships with this ADR. It formalizes the vocabulary the Phase 2C router docstrings already started (see Context) so implementation phases build one consistent model instead of four independent guesses.
- Alternatives considered:
  - *Fork the codebase per mode (separate Solo/Mobile/Shop builds).* Rejected — directly contradicts the required one-codebase model and reintroduces the exact "AI must never have a separate uncontrolled path" and duplicated-business-logic anti-patterns ADR-016 already forbids for the prompt/manual split; a third fork axis (mode) would be worse, not better.
  - *Encode mode as a tier attribute (e.g., "Solo tier implies Solo mode").* Rejected — conflates a workflow-shape decision (how the shop works day to day) with a payment decision (what the shop is entitled to); a shop must be able to change its operating mode without a billing event, and change its tier without a workflow disruption.
  - *Gate capabilities via a general-purpose feature-flag service (e.g., an experiment/rollout flag system).* Rejected for this decision — no such system exists in this codebase (§6 of the companion inventory confirms zero hits for "feature_flag"/"entitlement"/"capability" anywhere in `app/`), and a rollout-flag system solves a different problem (gradual exposure) than capability resolution (durable, shop-scoped, auditable state). A future pilot-controls/feature-flag phase (already tracked separately in `docs/context/GOAL_EVIDENCE_MATRIX.md` Part C, "Pilot controls/feature flags... Phase 11") may sit alongside this service later, but must not replace it as the authority for mode/tier capability.
  - *Rename the mode values to avoid the Solo/Shop word collision with tier names (e.g., `standalone`/`facility`).* Rejected as a default — the task's required mode vocabulary is Solo/Mobile Field/Shop/Technician verbatim, and inventing different internal names than the user-facing product language would itself be a form of "inventing names" the instructions rule out. The collision is instead flagged as Unresolved Decision #1 in the companion bridge document, to be resolved by an explicit owner call on internal field/enum naming (e.g., prefixing the column, like `operating_mode` vs `tier`, is already sufficient disambiguation at the schema level — the risk is in prose/logs/UI copy, not the schema).
- Consequences: every future phase that adds mode- or tier-aware behavior must go through the one capability service this ADR names, not a bespoke `if shop.tier == "solo"` check inline in a route or store function. The AST-based regression-test discipline already used to enforce ADR-019 (no raw `owner_user_id` comparisons) should be extended, at implementation time, to also fail the build on any raw `shop.operating_mode` or `subscription.tier` comparison outside the capability service — mirroring exactly how ADR-019's own enforcement mechanism works, per the same "one function computes the boundary, everything else calls it" principle.
- Files affected (when implemented): new `Shop.operating_mode` column + migration; new `app/capability_store.py` (or similarly named) capability-resolution service; new `ShopCapabilityOverride`/`ShopCapabilityEvent` tables (audited via the existing `ShopEvent` append-only pattern per `app/db_models.py:2232+`); a new `GET /api/capabilities` route; frontend nav wiring to consume it; and, when ADR-017 ships, the plan-executor's capability check. None of these exist yet.
- Revisit if: implementation (Phase-by-phase per the companion document's slice plan) discovers the single-service model cannot express a real requirement (e.g., a capability that depends on time-of-day or external system state in a way the resolve-once-per-request shape can't capture) — at that point this ADR should be revisited with the specific gap as context, matching ADR-019's own revisit discipline.
