# ADR-017: Prompt/manual shared execution

- ID: ADR-017
- Date: 2026-07-20
- Status: Accepted (design), Not yet implemented
- Context: The "one prompt → one plan → one approval → completed workflow" product principle does not exist anywhere in optimus-server yet — `app/orchestrator.py` is read-only research, and `app/services/optimus_chat.py`'s chat endpoint never touches the database. The Laravel PoC built and verified a working shape for this (`PlanGeneratorService`/`PlanApprovalService`/`PlanExecutorService`/`ReceiptService`), proven end-to-end via Playwright against a real browser and Postgres. Full detail in `docs/architecture/STACK-DECISION.md` §1.2 and §6.
- Decision: Build the equivalent shape directly on FastAPI: a plan-generator produces a structured, previewable plan (understood/missing-information/assumptions/proposed-actions/risk/confidence) with zero side effects; approval is one explicit action; execution calls the same store functions a manual UI click would call (per ADR-016), recording one audit event per action and halting the remaining chain on first failure; a receipt is assembled from those audit events. Re-approval is required only when the plan materially changes, pricing changes, parts are ordered, payment is captured, a refund is issued, customer data is exported, records are deleted, or another legally/financially distinct action is introduced — never for every individual internal step.
- Alternatives considered: building this feature in Laravel instead, since the PoC already has a working version (rejected per ADR-014 — this is net-new work regardless of target stack, so building it on FastAPI preserves the one-write-path guarantee without introducing a second language/runtime into the write path).
- Consequences: this is the first feature-level use of the security kernel (Phase 1) and the shared-service-layer discipline (ADR-016) together — its acceptance tests must include an approval-bypass test proving execution cannot occur without a prior, matching approval, for both the AI and manual paths.
- Files affected (when implemented): new `app/plan_store.py` (or similarly named) with `generate_plan()`/`approve_plan()`/`reject_plan()`/`edit_plan()`/`execute_plan()`, a new `Plan`/`PlanAuditEvent` schema and migration, new `APIRouter` endpoints for the Command Center (built on the router split from ADR-014), and frontend work for the collapsed-sidebar/Command Center UI.
- Revisit if: the approval-policy boundary (which actions require re-approval) needs to expand or contract based on real usage after this ships.
- Scope note: explicitly **not** part of Phase 1 (the security kernel). Phase 1 must complete and pass its own acceptance tests first, per the roadmap ordering in `docs/architecture/STACK-DECISION.md` §7.
