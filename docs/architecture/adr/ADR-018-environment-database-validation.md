# ADR-018: Environment/database validation

- ID: ADR-018
- Date: 2026-07-20
- Status: Accepted
- Context: The Laravel PoC's second architecture bug was a container correctly configured for PostgreSQL that silently served every HTTP request against SQLite instead, because `php artisan serve` re-read a checked-in `.env` file for the request-handling process and stripped the correct container environment variables. Nothing failed or logged an error; the mismatch was only found by comparing Postgres row counts by hand. Checked against optimus-server: this exact failure mode does not reproduce there — `uvicorn` has no equivalent env-reload-and-strip behavior, and `psycopg`/PostgreSQL is the only database driver installed at all (no sqlite dependency exists in `pyproject.toml`), so there is structurally no fallback path to a wrong database today. `GET /ready` (`app/main.py:896`) already performs real TCP reachability checks and a schema-compatibility comparison against the expected Alembic head, which is stronger than anything the PoC added.
- Decision: Add defense-in-depth against this class of failure (a correctly-configured environment silently degrading to a wrong one with no error) even though the specific mechanism doesn't apply here: assert at startup that `database_url` does not resolve to a non-Postgres driver when `APP_ENV` is `production`/`staging`, and extend `GET /ready`'s response with an explicit, non-secret `"database_driver": "postgresql"` field so the invariant is externally visible and monitorable (by a human or by Sentinel later), not just internally assumed.
- Alternatives considered: doing nothing, since the bug class is structurally prevented today (rejected — "structurally prevented by today's dependency list" is not the same as "tested and asserted," and a future refactor could reintroduce a fallback without anyone noticing until it's in production); building a Laravel-style entrypoint validation script (rejected — unnecessary, since FastAPI/uvicorn has no equivalent env-stripping behavior to guard against at the process-launch level; a boot-time assertion inside the app is sufficient).
- Consequences: low-priority, low-effort addition; should not block or be blocked by any other Phase 1 work, and can be folded into Phase 1's structured-error-capture extension or done as its own small slice.
- Files affected (when implemented): `app/main.py` (a startup assertion, and the `/ready` response field), a new test asserting the app refuses to start (or reports unready) when `APP_ENV` is production/staging and the configured driver is not postgres.
- Revisit if: a future local-dev convenience mode introduces a real sqlite fallback option, at which point this ADR's assumption ("no sqlite dependency exists") stops holding and the check becomes load-bearing rather than defense-in-depth.
