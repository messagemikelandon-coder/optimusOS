# OptimusOS Stack Decision — Architecture Decision Report

**Status:** Awaiting approval. No broad implementation has started.
**Scope:** Verifies and closes out the Laravel proof-of-concept phase; recommends a path forward; documents two architecture lessons the PoC surfaced; specifies the first implementation phase (a security kernel) for whichever path is approved.

### 1.2 Normalizing by feature coverage (the comparison the raw LOC ratio hides)

**1,148 PHP lines is not "60% cheaper" than 25,694 Python lines — it is a different amount of product.** The PoC implements exactly one vertical slice (customer → vehicle → estimate → plan → approval → work order → receipt) touching 4 of optimus-server's ~24 functional domains, and even within those 4 domains it implements a fraction of the real behavior:

| Domain | optimus-server depth | PoC depth |
|---|---|---|
| Estimates | Revisions, customer-facing approval tokens (send/approve/decline/expire/supersede/archive), snapshotting, web-research-backed pricing (`app/orchestrator.py` + `app/services/openai_web.py`), 1,092-line store | Flat tax-rate math, draft status only, no revisions or customer-facing approval link, ~45-line service |
| Vehicles | Tenant-scoped VIN uniqueness with archival support (`uq_vehicles_owner_active_vin`, partial index), search/pagination | Tenant-scoped VIN uniqueness **and** a full ownership-history table (see §4) — deeper than production in this one specific respect, shallower everywhere else (no archival, no search/pagination) |
| Work orders | Full status-transition state machine, technician assignment, parts allocation | Single `open` status, no state machine |
| Customers | Full CRUD, dedupe/archival, customer-history rollups across estimates/work orders/invoices | Find-by-phone-or-email-or-create only |

Domains with **no PoC equivalent at all**: scheduling/appointments/bays/technicians, invoices/payments/Square, purchase orders/parts/vendors/allocations, diagnostics, inspections, intake requests, notifications, dashboards/reports, shop/tenant/subscription billing/platform-support administration, account security/email verification, workflow gaps, customer-history rollups. That is roughly 20 of 24 domains.

**Extrapolated cost of true feature parity in Laravel:** if the PoC's 1,749 lines cover ~17% of domain breadth at a shallower depth than production in nearly every domain it touches, reaching actual parity (same breadth, same depth, same tenant/subscription/support model, same test rigor) plausibly requires **20–30× the PoC's current volume** — on the order of 35,000–50,000+ lines of PHP/Blade, a comparable multiplication of migrations, and a test suite of comparable scale to the 20,339 lines currently protecting the Python system. That is not a smaller system than optimus-server; it is a from-scratch rebuild of a comparably sized one, with none of the 65+ PRs' worth of accumulated bug fixes, and a schema that would need to re-derive the tenant/subscription/support model optimus-server just finished building and testing over the last two days.

This is the basis for treating "the PoC is smaller" as **irrelevant to the stack decision** — it was never testing efficiency, only feasibility of the pattern.

## 3. Recommendation

**A — Retain and simplify FastAPI/JavaScript, with two of the PoC's design lessons ported into it as targeted enhancements (§4, §5). Do not migrate to Laravel, phased or otherwise.**

This is not "FastAPI because it exists." Three things drove the matrix, all falsifiable and checked, not assumed:

1. **The application is not early or structurally unstable.** 65+ merged PRs, a disciplined fat-store/thin-route pattern applied consistently across 28 modules, a CI pipeline that rehearses backup/restore/rollback for real, and a tenant-isolation model that just passed a live conversion (11 migrations in 2 days, with an AST regression test that fails the build if any store reintroduces a raw `owner_user_id` comparison). None of the "recommend migration" conditions from the original decision rules are met.
2. **Normalized for feature coverage, the PoC does not show Laravel is cheaper — it shows the pattern (shared service layer, plan/approve/execute) is buildable in either stack.** Since that pattern is net-new work regardless of target, the deciding factor becomes what else you keep or discard, and Option A keeps everything; B and C discard tested, working code to re-derive it.
3. **The PoC surfaced two real bugs in under a day of work** (§4, §5) — a preview of the debugging cost a full rewrite would pay at 20–30× the surface area, on a system that has already paid down three weeks of exactly this kind of cost.

**Where the PoC earns its keep despite this:** its ownership-history model (§4) is a genuine improvement over what optimus-server has today, and its environment-validation pattern (§5) is worth adding as defense-in-depth even though the exact bug class doesn't reproduce in FastAPI. Both are recommended as scoped enhancements to the existing system, not as evidence for migrating to build them.

## 4. Architecture lesson 1 — Vehicle identity and ownership

### What happened in the PoC

`VehicleService::findOrCreateVehicle()` looked up an existing vehicle scoped to the *selected customer* (`where('customer_id', $customer->id)->where('vin', $vin)`), while the schema enforced VIN uniqueness *globally* (`unique()` on the bare column). Running the manual workflow for a second, different customer with a VIN already in use inserted a duplicate row and hit the database's unique-constraint violation — an unhandled exception, not a friendly "already exists" response.

### Root cause

Two different concerns were conflated into one mutable column: **vehicle identity** (which physical car is this) and **current ownership** (who owns it right now). A single `customer_id` foreign key can only express "the current owner," so any lookup or constraint built around it has to pick one of two wrong shapes: scope by customer (breaks when the same VIN legitimately needs a new owner) or don't scope by customer (breaks tenant isolation, since a global unique constraint would forbid two unrelated shops from ever recording the same physical car).

### Checked against optimus-server

optimus-server does **not** have the PoC's exact bug: `_ensure_unique_active_vin()` in `app/vehicle_store.py` already scopes the uniqueness check through `_owner_query()` (tenant-scoped) with a partial index (`uq_vehicles_owner_active_vin`, `WHERE vin IS NOT NULL AND is_archived = 0`) that also correctly excludes archived vehicles. This is more sophisticated than the PoC's first draft.

What optimus-server **does** lack: any ownership-*reassignment* capability at all. `VehicleUpdate` (`app/models.py:785`) does not include `customer_id` as an updatable field — a vehicle's owner is fixed permanently at creation. There is no `previous_owner`/history concept, and no supported way to record "this car was sold" without archiving the vehicle and losing its service-history linkage. This is an absent capability, not a live bug, but it is the same gap the PoC's fix closes.

### Long-term model (decided)

- **Identity**: a VIN identifies one vehicle *within its tenant boundary* — uniqueness is `(shop_id, vin)`, never global. Two unrelated shops may each have their own record of the same physical car at different points in its life; that is not a conflict.
- **Ownership**: a separate `customer_vehicles` history table, not a replaceable `customer_id` column. Exactly one row per vehicle may be "open" (`effective_to IS NULL`) at a time — the current owner — enforced by a partial unique index at the database level, not just application logic. Every prior owner and the date range they held the vehicle remains queryable and is never deleted or overwritten.
- **Reassignment rule**: `reassignOwnership(vehicle, newCustomer)` closes the current open row (`effective_to = now()`) and opens a new one; it is a no-op if the new owner is already current. Every reassignment is audited (`vehicle.ownership_reassigned`, old/new customer IDs, vehicle ID) regardless of whether it was triggered by a manual click or a plan execution.
- **Estimate/work-order attribution**: an estimate keeps the `customer_id` it was created under permanently. That attribution does not change when the vehicle is later reassigned — a job authorized by the original owner stays attributed to them even after the car is sold. Full service history (all estimates/work orders for a vehicle) is queried by `vehicle_id`, independent of current ownership.

### Verification

Implemented in the PoC (`app/Domain/Vehicles/VehicleService.php`, migrations `2026_07_20_053304`/`2026_07_20_053305`) and proven by `tests/Feature/VehicleOwnershipTest.php` (6 tests, all passing):
same VIN across two shops is allowed; same VIN within one shop dedupes instead of duplicating; reassigning ownership preserves history and prior estimate attribution; reassignment is audited; the database itself (not just application code) rejects a duplicate `(shop_id, vin)` and a second open ownership row, via a real `QueryException` in both cases.

### Recommendation for optimus-server

Add a `vehicle_customers` (or similarly named) ownership-history table with the same effective-dating shape, and a `reassign_vehicle_owner()` store function that closes/opens rows and emits a `VehicleOwnershipEvent`, mirroring the existing `EstimateApprovalEvent`/`WorkOrderStatusEvent` audit-trail pattern already used elsewhere in the codebase. This is additive (new table, new function, no changes to existing `Vehicle.customer_id` reads) and should be scheduled as ordinary backlog work, not urgent — there is no current data-loss bug, only a missing capability.

**Status per this preservation: NOT implemented in this phase.** Per the approved directive, this is tracked as a separate ADR/migration plan (`docs/architecture/adr/ADR-015-vehicle-identity-and-ownership.md`) and is explicitly out of scope for the Phase 1 security-kernel work.

## 5. Architecture lesson 2 — Environment/database validation

### What happened in the PoC

The `docker-compose.yml` `app` service was correctly configured with `DB_CONNECTION=pgsql` as a container environment variable, confirmed via `docker compose exec app env`. Yet Postgres queried directly via `psql` showed zero rows after browser-driven end-to-end tests reported success. Root cause, found by adding a temporary debug route: Laravel's `php artisan serve` command re-reads the checked-in `.env` file (which said `DB_CONNECTION=sqlite`) for the child process that actually serves HTTP requests, and by design strips most of the parent process's environment variables from that child unless `--no-reload` is passed or no `.env` file exists (`Illuminate\Foundation\Console\ServeCommand::startProcess()`, `shouldPassThroughEnvironmentVariable()`). Nothing failed. No error was logged. The application simply served every request against the wrong database, and the only way to detect it was to compare row counts by hand.

### Why this matters beyond the PoC

The severity here isn't the specific Laravel quirk — it's the failure mode class: **a correctly-configured environment silently degrading to a wrong one, with no error, no log line, and no way to detect it short of manually cross-checking data.** That class of bug is worth guarding against regardless of which stack is chosen.

### Checked against optimus-server

The exact bug does not reproduce there: uvicorn has no `.env`-reload-and-strip behavior, and `psycopg`/postgres is the only database driver installed at all (`pyproject.toml` has no sqlite dependency) — there is structurally no fallback path to a wrong database today. `GET /ready` (`app/main.py:896`) already does real TCP reachability checks against Postgres/Redis and a schema-compatibility comparison against the expected Alembic migration head, which is a stronger check than anything added in the PoC. optimus-server does not need the PoC's specific fix; the general principle is still worth applying as defense-in-depth against a future refactor (e.g., a hypothetical local-dev sqlite convenience mode) accidentally reopening this class of risk.

### Design (implemented in the PoC, recommended as a pattern generally)

- **Startup validation**: `DatabaseEnvironmentGuard::assertSafe(environment, driver)` throws immediately if an environment in a disallowed list (`production`, `staging`) is configured for a disallowed driver (`sqlite`). Scoped to HTTP boot only (not console) so build-time tooling (`composer install`'s `package:discover`, migrations, tinker, the test suite) is never blocked by a check meant to catch real traffic being served wrong.
- **Container entrypoint rule**: `docker/entrypoint.sh` runs `php artisan app:validate-database-environment` before the server starts at all; a misconfigured container refuses to boot rather than serving one bad request. Verified directly: `docker run --rm -e APP_ENV=production -e DB_CONNECTION=sqlite optimus-laravel-poc-app` exits 1 with a clear error before any request is served.
- **No checked-in `.env` in the image**: `.dockerignore` now excludes `.env` from what gets baked into the image at all, closing the specific root cause (a stale file being read at runtime) rather than only papering over its symptom.
- **Non-secret health diagnostic**: `GET /health/database` returns `{"ok": true, "environment": "local", "driver": "pgsql", "reason": null}` — driver and environment name only, never host, database name, username, or password (verified by a dedicated test asserting those keys are absent from the response). This makes a wrong database visible from outside the container instead of only discoverable by comparing row counts after the fact.
- **SQLite remains fully legitimate** for local development and the fast in-memory test suite (`phpunit.xml` sets `DB_CONNECTION=sqlite`, `DB_DATABASE=:memory:` for `APP_ENV=testing`, which the guard explicitly allows) — this is not a ban on sqlite, only a ban on it silently reaching production or staging traffic.

### Verification

`tests/Feature/DatabaseEnvironmentGuardTest.php` (8 tests, all passing): production+sqlite rejected, staging+sqlite rejected, production+pgsql allowed, local+sqlite allowed, testing+sqlite allowed, `assertSafe()` throws for the disallowed combination, the health route reports the driver without leaking any of the four checked secret-shaped keys, and the artisan command exits code 1 when misconfigured. Additionally verified live: `docker compose build` now fails loudly with this exact guard's error message if the check accidentally runs at build time (it originally did — the guard caught its own over-broad first draft during `composer install`, which is why it's now scoped to HTTP-only boot), and the full stack was rebuilt, brought up, migrated, and re-driven through both Playwright e2e tests successfully afterward with `/health/database` confirmed reporting `pgsql`.

**Status per this preservation:** the general "fail loud, never silently serve the wrong database" principle is folded into Phase 1's acceptance tests (`docs/architecture/adr/ADR-018-environment-database-validation.md` and `docs/architecture/PHASE1-SECURITY-KERNEL-PLAN.md`) as a startup-config check, since optimus-server already has no sqlite fallback path in production code but currently has no explicit test proving that invariant holds.

## 6. Architecture decision records

### ADR-001: Final stack and deployment

**Decision:** Retain FastAPI + SQLAlchemy + PostgreSQL + Alembic as the canonical backend, and vanilla JS + Blade-equivalent server-rendered patterns are not adopted — the existing static-JS frontend is kept and incrementally improved. Do not introduce Laravel/Livewire/PHP into the production deployment.
**Simplification applied in the same breath:** collapse the 5-container deployment to 3 by (a) having FastAPI serve `app/static/` directly (it already mounts `/static`) instead of a separate nginx container, and (b) removing the `worker` service, which today is verified to do nothing but a 60-second TCP heartbeat loop — no queue is consumed, nothing is enqueued to it anywhere in the codebase.
**Consequence:** `app/main.py`'s 178 endpoints get split into `APIRouter` modules matching the existing 28-store grouping — a mechanical, zero-behavior-change refactor that directly serves this decision by giving the new Command Center/plan-executor endpoints (ADR-004) an obvious small home instead of a 29th addition to one 5,105-line file.

### ADR-002: Vehicle identity and ownership

**Decision:** Adopt the PoC's model in optimus-server: vehicle identity is `(shop_id, vin)` (already correct there), and ownership becomes a `vehicle_customers`-style history table with effective dating, replacing the current permanent, non-reassignable `customer_id` — see §4 for the full design and verification. This is additive backlog work, not urgent, since no data-loss bug exists today.

### ADR-003: Shared business-service layer

**Decision:** No change needed to the pattern — optimus-server's `*_store.py` modules already are the shared service layer the product principle requires ("prompt and manual interfaces must use the exact same application services"). The PoC independently re-derived the identical shape (`Domain/*Service.php`, one method per business operation, called identically from Livewire UI and the plan executor), which is the strongest available evidence that this is a sound, stack-independent pattern rather than a Laravel-specific one. The one discipline to enforce going forward: the AI plan-executor (ADR-004) must call existing store functions and must never reimplement `effective_shop_id()`/`_owner_query()` scoping itself.

### ADR-004: Prompt/manual shared execution

**Decision:** Build the "one prompt → one plan → one approval → completed workflow" feature directly on FastAPI, following the PoC's proven shape: a plan-generator produces a structured, previewable plan (understood/missing-info/assumptions/proposed-actions/risk/confidence) with no side effects; approval is a single explicit action; execution calls the same store functions manual UI calls, recording an audit event per action and halting the remaining chain on first failure; a receipt is assembled from those audit events. This is net-new work regardless of stack (§1.2) — building it on FastAPI preserves the one-write-path guarantee without introducing a second language/runtime into the write path.

### ADR-005: Environment/database validation

**Decision:** Add a startup assertion to optimus-server mirroring `DatabaseEnvironmentGuard`'s intent even though the specific bug class doesn't reproduce there today: assert at boot that `database_url` does not resolve to a non-Postgres driver when `APP_ENV` is `production`/`staging`, and extend `GET /ready`'s existing response with an explicit `"database_driver": "postgresql"` field (driver name only, already-non-secret) so the invariant is externally visible and monitorable by Sentinel, not just internally assumed. Low priority — defense-in-depth for a risk that is currently structurally prevented by the absence of a sqlite dependency, not an active gap.

### ADR-006: Tenant boundary

**Decision:** No change — `effective_shop_id(db, auth)` remains the single tenant-scoping primitive, reused by every store and now additionally validated by the PoC's independent confirmation that "one function computes the boundary, everything else calls it" is the right shape (the PoC's `shopId` parameter threading through `VehicleService`, though far less developed, follows the identical principle). The AST regression test that already fails the build on any raw `owner_user_id` comparison stays as the enforcement mechanism.

### ADR-007: Security-kernel integration

**Decision:** Before any Sentinel autonomy work begins, extract a reusable security kernel (§8) covering authentication, authorization/roles, tenant isolation, audit logging, approval policies, API-key controls, rate limits, input validation, file access, secret redaction, and structured error capture. Existing pieces (`app/auth.py`'s `effective_shop_id`, `app/rate_limit.py`'s Redis sliding-window limiter, `app/security_events.py`'s event enum, `app/security.py`'s Argon2id hashing) are consolidated and hardened rather than rewritten — this phase is explicitly scoped as extraction-and-hardening, not new invention, since all the pieces already exist and work.

### ADR-008: Sentinel event integration

**Decision:** Sentinel integrates through structured events, middleware, and registered playbooks — not premature microservices, and not autonomous action. Initial scope is strictly observe/organize/correlate/alert/recommend. It must never execute arbitrary AI-generated shell commands or modify production state outside a registered, human-approved playbook. This ordering (security kernel first, Sentinel observation-only second, Sentinel action only behind registered playbooks third) is a hard sequencing constraint, not a suggestion — see §7.

## 7. Ordered roadmap

1. **Approval of this report** (current step — no further implementation until this is reviewed).
2. **Security kernel** (§8) — extraction and hardening of existing auth/authz/tenant/audit/rate-limit/validation code into one reusable module. Nothing in steps 3+ begins before this ships and passes its own acceptance tests (§10).
3. **Deployment simplification** (ADR-001) — drop the `worker` and `frontend` nginx containers, serve static files from FastAPI directly, split `main.py` into routers. Independently valuable, low-risk, no behavior change; can run in parallel with step 2 if capacity allows, but must not be blocked on or blocking it.
4. **Vehicle ownership history** (ADR-002) — additive `vehicle_customers` table and reassignment function. No urgency; schedule as ordinary backlog once steps 2–3 are stable.
5. **Prompt/manual shared execution** (ADR-004) — the Command Center, plan generator/approval/executor, and receipt UI, built on the now-hardened security kernel and the existing store layer.
6. **Environment/database validation hardening** (ADR-005) — extend `/ready`, add the boot-time assertion. Low effort, can be folded into step 2 or done as a small standalone slice.
7. **Sentinel — observation only** (ADR-008) — structured event emission (tenant/user/request/workflow/action/result/error/deployment-version/timestamp) from every store function and the new plan executor. No autonomous action capability exists yet at the end of this step.
8. **Sentinel — registered playbooks with approval** — only after step 7 has run in production long enough to validate its event data is trustworthy. Any action capability is gated by a registered playbook and an explicit human approval per the same one-approval-per-plan model as step 5, never arbitrary AI-generated commands.

## 8. First security phase — exact scope

This is the first implementation phase once this report is approved. All items below already exist in some form in optimus-server; this phase extracts, consolidates, and hardens them into one reusable module rather than inventing new mechanisms.

| Capability | Existing code to extract from | Hardening work |
|---|---|---|
