## Historical Resolved Issues

- Chart.js metric-card sparkline layout bug found and fixed during the Overview dashboard build (2026-07-11, resolved): `<canvas class="metric-sparkline" width="120" height="30">` elements were passed straight to `new Chart(canvas, { options: { responsive: true, maintainAspectRatio: false } })` with no CSS-sized parent container. Chart.js's responsive resize logic ignores the canvas's own `width`/`height` attributes once `responsive: true` is set and instead sizes to its parent — with no parent height constraint, the canvas grew unbounded, which stretched the entire `.metric-grid` row (CSS Grid rows size to their tallest cell) to several thousand pixels tall. Caught live via a real Playwright screenshot against real seeded data (not caught by any static check, since `node --check`/tests don't render layout). Fixed by wrapping each sparkline canvas in a new `.metric-sparkline-frame` div (`position: relative; height: 30px`) with `canvas { width/height: 100% !important }`, matching the pattern already used for the two full-size trend charts' `.chart-frame` wrapper. Re-verified live after the fix: normal card heights, charts render correctly at both desktop and mobile viewports.
- Two latent CSP violations found and fixed during the landing-page redesign (2026-07-10, resolved): (1) an inline `<script>` bootstrap block added to `app/static/index.html` during this same session would have been silently blocked by the app's own `script-src 'self'` CSP (`app/main.py::security_headers`) in any real, non-proxied request to the backend — caught by independent review before it was ever exercised live, then confirmed with a live Playwright check directly against the rebuilt backend container (port 8000, which does apply the CSP, unlike the local dev nginx frontend at port 5173 which serves `app/static` via a bind mount and doesn't route "/"/"/login"/"/approval" through the backend at all). Fixed by moving the logic into the existing pathname branch in `app/static/app.js::initializeApp` instead. (2) That same live CSP check also surfaced a **pre-existing, unrelated** `style-src 'self'` violation on every single page load: `app/static/index.html`'s Square dashboard panel had a `style="grid-column: 1 / -1;"` inline attribute (from the earlier Square feature slice) that had never been checked against the real CSP before. Fixed by replacing it with a new `.square-panel-full` class in `styles.css`. Regression tests added: `test_index_html_has_no_inline_scripts` and `test_index_html_has_no_inline_style_attributes` in `tests/test_official_ui.py`, so neither class of violation can silently reappear. Both fixes verified live against the real backend (port 8000) with zero CSP console violations after rebuild.
- `scripts/optimusctl.sh` staging-override gap (2026-07-09, resolved): the script previously ignored `ops/docker-compose.staging.yml`, so running `update`/`restart` on the staging droplet would have silently dropped the public `0.0.0.0:80` binding back to the dev-only `127.0.0.1` default and taken the site offline. Fixed in `7d665c8`: the script now reads an optional `COMPOSE_OVERRIDE_FILE` key from `.env` and appends it to every compose command (absent key = local dev byte-identical; missing file = hard fail). Verified via sandboxed command-assembly xtrace, merged-config port comparison, and a real local `status` run.
- Staging owner password rotation after the second leak incident (2026-07-09, resolved): owner confirmed the password was rotated to a genuinely new, never-reused value.
- Invoice detail buttons ("Open work order," "Open HTML," "Open PDF") staying disabled bug (2026-07-09, resolved): root-caused to an async race in `app/static/app.js` between an unawaited `loadInvoices()` (fired by `navigate("invoices")`) and an awaited `selectInvoice()` — a slower, stale list response could wipe a newer, correct selection back to null. Fixed with a `state.invoices.selectionVersion` monotonic counter so the newer selection always wins. Live-tested with Playwright network-interception to deterministically reproduce the bug on the original code and confirm the fix; full gate suite green. Committed `1139499` on `ops/staging`. A separately reported concern ("Create work order disabled after saving an estimate") was investigated and found to be correct, by-design behavior (requires full estimate approval, not just saving) — not a defect.
- Secret exposure incident #1 (2026-07-09, resolved): running `docker compose config` during Phase 5 staging-override validation fully resolved and printed the real local `.env` contents (including `OPENAI_API_KEY`, `OPTIMUS_OWNER_PASSWORD`, `OPTIMUS_ACCESS_TOKEN`) into the session transcript — `docker compose config` does not filter secrets, and the command wasn't piped through a filter before its output was read. Disclosed to the owner immediately. Owner rotated all three values same-day; local `backend`/`worker` containers force-recreated to load the new values. No commit, log file, or persistent artifact contains the leaked values. Going forward: avoid full `docker compose config` dumps entirely when validating Compose overrides; reason through YAML merge semantics directly instead (as already done successfully for the Phase 4 e2e override).
- Secret exposure incident #2 (2026-07-09, disclosed, rotation not independently confirmed): while setting the staging owner password on the droplet, repeated terminal/paste mangling (auto-indent in `python -c` and `nano`, then a heredoc `EOF` terminator that never matched) led to the owner's chosen staging password (`loomis4you` — the same value leaked in incident #1) being typed into a stuck shell prompt and appearing in the session transcript. Owner was told to treat it as compromised and pick a new, not-reused value; this was not independently re-confirmed before the session moved on. Root cause of the terminal mangling itself: the droplet's console does not handle multi-line paste cleanly for any of `python -c`, `nano`, or heredocs. Workaround that reliably worked: build files via a sequence of single-line, independently-complete `echo '...' >> file` commands instead. Separately, also discovered: editing `.env` on disk does not update an already-running container's environment — the owner password reset only took effect after `docker compose up -d --force-recreate backend worker`.

- Phase 5 (Private Staging) local-prep review items resolved on 2026-07-09:
  - independent review found `tag_current_as_previous()`/`rollback()` in `scripts/optimusctl.sh` called bare `docker image inspect`/`docker tag` instead of going through the script's existing sudo-detection wrapper (`COMPOSE`), which would break on any host needing `sudo docker` and would have regressed the existing `update` command too; fixed by adding a matching sudo-aware `DOCKER` array and routing both functions through it
