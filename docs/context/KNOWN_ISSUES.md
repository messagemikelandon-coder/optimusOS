- **Request-level concurrency: this app currently processes at most one HTTP request at a time, system-wide, regardless of load** (found 2026-07-17 while adding `tests/e2e/test_scheduling_concurrency.py`, confirmed empirically not just by code reading). Every route in `app/main.py` is declared `async def` but calls directly into a synchronous, blocking store function (e.g. `create_appointment`, `create_customer`) without offloading it to a thread pool (`await run_in_threadpool(...)`/`asyncio.to_thread(...)`) — a blocking call made directly inside an `async def` coroutine blocks the entire single-threaded `asyncio` event loop for its duration. Combined with the real `Dockerfile`'s `CMD` (`uvicorn app.main:app ...`, no `--workers` flag — a single worker process), this means two genuinely simultaneous incoming requests are not actually processed concurrently: the second request's handler cannot even begin running until the first request's entire blocking DB round-trip completes. Confirmed by timing two "concurrent" HTTP requests against a real running instance with a deliberate 1-second delay inserted into one store function: they took ~2s total, not ~1s. This is why an early version of the Scheduling concurrency test (driving two requests through real HTTP) passed identically whether or not the DB-level row lock was even present — there was never a real race for it to prevent at the HTTP layer, since the app's own request handling already fully serializes everything. Not a data-integrity bug (nothing is unsafe today) and not fixed here — it's a real throughput/scalability ceiling worth knowing about before assuming this app can serve concurrent real users at any real deployment scale, and the reason `tests/e2e/test_scheduling_concurrency.py` calls `scheduling_store.create_appointment` directly (bypassing HTTP, using two real threads with two real Postgres connections, which do genuinely run concurrently since the GIL releases during real socket I/O) rather than through the live server. Revisit if multi-user concurrent load ever becomes a real requirement — the fix is either wrapping every route's synchronous store call in `run_in_threadpool`, or migrating to an async DB driver end-to-end, or running multiple `uvicorn` workers (which would also require re-examining every in-process singleton this codebase uses, e.g. the rate limiters' in-memory fallback and `tests/conftest.py`'s autouse reset fixture, none of which would work correctly split across multiple worker processes without a shared store like Redis).

- SQLite does not enforce foreign keys (including every `ON DELETE CASCADE` in the schema) unless a per-connection `PRAGMA foreign_keys=ON` is explicitly issued, and `app/db.py::build_engine` does not do this — discovered 2026-07-13 while building Phase 6 Part B's synthetic-account cleanup, when a test proved a deleted owner's linked technician `UserAccount` row was *not* actually removed under the test suite's SQLite engine, even though the exact same operation correctly cascades against real Postgres (verified live in this same session). A global fix (enabling the pragma for every SQLite connection) was attempted and reverted the same session: it broke roughly 100 pre-existing tests across most of the API test files with `FOREIGN KEY constraint failed` errors, almost certainly reflecting FK-insertion-order assumptions baked into fixtures/helpers across the whole suite that have never been exercised under real constraint enforcement — that is itself a real, wider-reaching gap, but fixing it is a properly-scoped task of its own, not something to fold into an unrelated feature slice. Worked around locally for Part B by making its own cleanup logic (`app/test_support_store.py::_delete_owner_and_dependents`) explicitly delete linked technician rows rather than relying on passive DB-level cascade — correct and verified on both engines. Every *other* cascade relationship in the schema (owner → customers/vehicles/estimates/work orders/invoices/etc.) still only has test coverage that passes on SQLite without truly exercising the FK cascade path; it is proven correct only against real Postgres in prior live-proof sessions, not by the automated test suite itself. Worth a dedicated pass (enable the pragma, then fix each fixture/test that assumed lax ordering) before this is considered fully verified.
- Reports module, Technician Time report (Phase 6 Part G slice 1, `app/report_store.py::get_technician_time_report`, 2026-07-15): the report filters time entries on `clock_in_at` falling inside the requested window, not on any clock-in/clock-out *overlap* with the window. A shift that clocked in before `date_from` and clocked out inside the window has none of its in-window hours counted — not just the portion before the window, all of it. Flagged by independent review; accepted as-is since shifts rarely span a reporting-window boundary in this shop's usage, but not fixed. Revisit with an overlap-based filter if boundary-spanning shifts turn out to matter in practice.
- Dashboard Gross Profit / Gross Profit Margin, "usage-period" COGS approximation (Phase 6 Part G slice 2a, `app/dashboard_store.py::_period_cogs`, 2026-07-16): there is no FK linking invoice line items back to the `Part` catalog, so COGS can't be matched to exactly the revenue it corresponds to (proper accrual accounting). Instead, the cost side sums `PartAllocation.unit_cost_snapshot * quantity` for `PartAllocationEvent` rows with `event_type='used'` whose `created_at` falls in the requested window — i.e. parts *used* in the period, not parts whose *revenue was invoiced* in the period. A part used just before a window boundary but invoiced just after (or vice versa) will show its cost and its revenue in different reporting periods. Deliberate, disclosed approximation (not a bug): parts usage and invoicing happen close together in this shop's actual workflow, and the alternative (refusing to compute Gross Profit at all, as it was hardcoded before this slice) was judged worse than a labeled approximation. Also: part-usage quantities with no `unit_cost_snapshot` recorded are excluded from the COGS dollar sum entirely (not assigned a fabricated cost) and surfaced separately via a `parts-missing-cost-data` dashboard insight. Revisit if the business ever needs period-exact accrual accounting (would require either an invoice-line-to-part FK or point-of-sale cost capture). The new Parts Usage report (Phase 6 Part G slice 3, `app/report_store.py::get_parts_usage_report`) shares this same underlying `PartAllocationEvent` data source and disclosure pattern per-part, but uses this file's `date_to <` (exclusive) convention rather than `_period_cogs`'s `<=` (inclusive), so the two reports' totals for a nominally identical date range won't always tie out to the penny at the boundary — documented in both functions' docstrings.
- Vendor Purchasing report, partially-received-then-cancelled spend exclusion (Phase 6 Part G slice 3, `app/report_store.py::get_vendor_purchasing_report`, 2026-07-16): a purchase order can be cancelled from `partially_received` status, meaning some of its parts were genuinely received (and paid for) before the remainder was cancelled. `PurchaseOrder.total` is fixed at creation time and isn't reduced by partial receiving, so there's no reliable already-spent subtotal to report — the report excludes the *entire* order's total from `total_spend` in this case, same as a fully-unreceived cancellation, rather than fabricating a partial figure. The exclusion is at least surfaced via `cancelled_order_count` rather than silently vanishing. A fully `received` order can never be cancelled afterward (RECEIVED is a terminal status), so this gap is narrower than it sounds — only reachable via partial receiving followed by cancellation. Pinned by `tests/test_reports_api.py::test_vendor_purchasing_report_partially_received_then_cancelled_excludes_full_total` so the current (imperfect) behavior can't silently change unnoticed. Revisit if this pattern turns out to matter in practice (would require tracking spend per received line item rather than per order).
- `tests/e2e/test_core_workflow.py` (Phase 6 Part C, 2026-07-13) showed real browser-timing flakiness during development: `page.select_option()` waits for the `<select>` element to exist but not for the *specific option value* to be present, and `#work-order-next-status` in particular is rebuilt by a second, fire-and-forget re-render shortly after each status transition, which could silently reset a just-made selection back to empty (the app then no-ops with a toast instead of erroring — surfaced as a `TimeoutError` waiting for a network response that never fired, not a failed assertion, making it easy to misdiagnose as a hung server). Fixed with an explicit select-verify-retry pattern on both `#work-order-next-status` and `#vehicle-customer-id` (the two dropdowns populated by an async fetch after another action). Stable across the last 4 consecutive local runs after both fixes, but real UI-timing races are never conclusively proven eliminated by a handful of runs — if this test flakes in CI, don't assume it's a new bug before checking whether it's the same class of race recurring elsewhere in the page.
