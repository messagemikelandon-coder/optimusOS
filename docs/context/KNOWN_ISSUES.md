## Recently Resolved (2026-07-04, Estimate Approval repair)

A controlled live proof of the Estimate Approval slice found three confirmed defects, repaired this session and verified with a non-billable Playwright run against the rebuilt Docker stack (exit code 0, run twice) plus 120 passing pytest tests:

- **Approval link routing**: `navigate()` in `app/static/app.js` cleared `window.location.hash` when switching to the approval view, so generated `/approval#token=...` links resolved to `/` on direct navigation and on refresh. Fixed by preserving the hash in the `history.replaceState` call. Regression introduced in commit `d162a55`.
- **Public approval-view data exposure**: `POST /api/estimate-approval/view` (unauthenticated, token-scoped) returned the full internal estimate/research payload, including unselected competitor part options/pricing, internal labor reasoning, and raw rate/fee overrides, to anyone holding a valid approval link. Fixed by adding narrow customer-safe response models (`EstimateApprovalPublicView` and related, in `app/models.py`) and wiring them into `get_approval_view()` in `app/estimate_store.py`. Confirmed by field-by-field review and a payload-shape regression test.
- **Fabricated zero-hour labor line**: `_validate_generated_estimate()` in `app/estimate_store.py` only rejected an estimate when both labor and parts were hollow, so a non-empty `labor_items` list whose lines all collapsed to zero hours/total could pass validation next to real parts pricing, showing the customer a nonsense free-labor line. Tightened to reject that case while still accepting legitimate labor-optional (parts-only) and parts-optional (labor-only) jobs.

A fourth suspected defect (one generation action allegedly triggering two OpenAI Responses API calls) was investigated and found to already be handled correctly by the existing model-fallback loop in `app/services/openai_web.py` (retries only on genuine model-unavailability). Closed with new regression tests proving call counts; no production code change was needed.

## Resolved (2026-07-06): estimator_output_invalid schema mismatch

**Root cause** (confirmed via source-level tracing of the installed `openai` Python SDK 2.44.0, not guessing): `OpenAIWebResearchService._structured_request()` called `client.responses.parse(text_format=CombinedResearchEnvelope, ...)`. The SDK's `.parse()` eagerly deserializes the model's raw JSON output text via `CombinedResearchEnvelope.model_validate_json(text)` *inside* the SDK (`openai/lib/_parsing/_responses.py::parse_text`), invoked as an unguarded `post_parser` callback that sits outside the SDK's own response-validation exception handling (`openai/_response.py:326-328`, which only wraps the earlier top-level envelope construction, not this later step). If that strict, eager validation fails for any reason — confirmed reproducible when the model returns `null` for optional fields such as `labor.basis`, `labor.special_tools`, `labor.risk_flags`, `parts.notes`, or `warnings` when it has nothing to report for them — the `pydantic.ValidationError` propagates out of `.parse()` with **no `Response` object attached at all**. That made the service's own lenient fallback chain (`_response_output_text` → `parse_json_object`/markdown fallback → `_coerce_combined_research_envelope`, which already tolerated exactly this shape) **unreachable** for this failure class: it only ran when `.parse()` succeeded but merely returned `output_parsed=None` (e.g. a refusal), never when `.parse()` itself raised. The test suite's fake OpenAI client hand-built an already-valid `CombinedResearchEnvelope` object directly and never exercised the SDK's real eager-validation step, so this was never caught locally before reaching the live proof.

**Fix**: `app/services/openai_web.py`'s `_structured_request()` now calls `client.responses.create(...)` instead of `.parse()`, manually building the identical OpenAI-side structured-output schema via a new `_text_format_param()` helper (confirmed byte-identical to what `.parse()` built internally — same `type_to_response_format_param` helper, just reshaped). This keeps the request the model receives completely unchanged (same schema constraint, same one API call), but guarantees the real raw `Response` object is always available, so the response is always processed through the service's own existing, already-tested lenient adapter (`_response_output_text` → `parse_json_object`/`_parse_markdown_research_fallback` → `_coerce_combined_research_envelope`) rather than the SDK's fragile, all-or-nothing convenience path. No validation was weakened, no prices were hard-coded, and no second OpenAI call was added — this was a response-handling change only.

**Proof**:
- `tests/test_openai_research.py::test_null_optional_research_fields_reproduce_and_are_fixed` first proves the exact shape raises `pydantic.ValidationError` under strict, eager validation (documenting the "before" failure), then proves the repaired adapter parses the identical payload successfully via a realistic fake client — 10/10 tests in the file pass, 120/120 in the full suite.
- A local, non-network reproduction script exercised 9 realistic-but-imperfect payload shapes (numeric strings, integers where decimals expected, null optionals, empty arrays, extra unknown fields, missing pricing, malformed JSON, narrative-only text) against both the old strict-validation path and the new adapter; only the null-optional-fields shape reproduced the live failure, and the new adapter now handles it (and every other shape) correctly, while malformed JSON and narrative-only text are still correctly rejected end-to-end.
- Non-billable Docker-level runtime proof (Phase 6): with `OPENAI_BASE_URL` temporarily pointed at a local mock server (removed afterward; `.env` diff confirmed clean; backend/worker restarted to restore real API access), the real, unmodified application code path (real FastAPI route → real `OptimusResearchOrchestrator` → real `OpenAIWebResearchService._structured_request`, i.e. exactly the fixed code) was exercised end-to-end through the actual browser-driven frontend: authenticate → create synthetic customer/vehicle → generate a real "Front brake pad and front brake rotor replacement" estimate against the mock (which deliberately returned the null-optional-fields shape) → confirmed exactly one request reached the mock and the adapter parsed it correctly (pad + 2 rotors + server-calculated total, no hard-coded prices) → reload from PostgreSQL → send for approval → open the real generated approval link → confirmed full customer-facing rendering with no supplier-cost/markup/margin leakage → approve → confirmed approval persistence and estimate locking → restart backend and worker → confirmed estimate and approval survived the restart. Log inspection found no secrets, session cookie values, raw model output, or tracebacks.
- The exact live-incident payload was never captured (by design, to avoid logging raw model output), so it cannot be proven that null-optional-fields was the *only* trigger — but it is a confirmed, reproducible member of the same failure class the fix eliminates, and the fix is architectural (makes the adapter universally reachable) rather than a narrow patch for one specific field.

**Files changed**: `app/services/openai_web.py` (the fix), `tests/test_openai_research.py` (rewritten fake OpenAI client to mock `.create()` instead of `.parse()`, matching the new call; two new tests; net test count in this file unchanged at 10 since two now-redundant tests describing the old `.parse()`/`output_parsed` distinction were retired).
